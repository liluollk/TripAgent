# Trip Agent 问题记录

## 2026-05-31 - Agent 系统调试修复

### 问题 1: DeepSeek API 超时 (ReadTimeoutException)
- **现象**: API 调用约 11 秒后超时
- **原因**: 手动创建 DeepSeekApi bean 时，HTTP 客户端超时配置未生效
- **解法**: 同时配置 RestClient（同步）和 WebClient（响应式）的超时时间为 180 秒

### 问题 2: Jackson 反序列化失败
- **现象**: Plan 类无法反序列化，缺少默认构造函数
- **解法**: 给 Plan、PlanStep、StepResult 类加上 @NoArgsConstructor 和 @AllArgsConstructor

### 问题 3: 枚举值不匹配
- **现象**: LLM 返回 "ATTRACTIONS" 但枚举值是 "ATTRACTION"
- **解法**: 在 parsePlan() 中添加 JSON 正则替换

### 当前状态
- ✅ Agent 完整流程跑通：规划 → 执行 → 结果
- ✅ SSE 流式输出正常
- ✅ 所有 MCP 工具正常工作（天气、景点、酒店、餐厅）

## 2026-05-31 - 代码质量审查与修复

### 问题 4: TripAgent 异步竞态条件和资源泄漏
- **现象**: 嵌套 subscribe() 导致 fire-and-forget，步骤完成判断用 list.size() 有竞态风险
- **原因**: 不正确的 Reactor 异步编程模式
- **解法**: 用 Flux.concatMap 顺序执行步骤，flatMapMany 链式调用替代嵌套 subscribe

### 问题 5: ReActLoop 阻塞 Reactor 线程
- **现象**: Flux.create 内部调用同步阻塞的 chatModel.call()
- **原因**: 阻塞调用在 Reactor 调度线程上执行
- **解法**: 用 Mono.fromCallable + subscribeOn(Schedulers.boundedElastic()) 隔离阻塞调用

### 问题 6: ToolRegistry JSON 注入漏洞
- **现象**: wrapInputToJson 用字符串拼接构建 JSON
- **原因**: 只转义了双引号，未处理反斜杠、换行等特殊字符
- **解法**: 改用 ObjectMapper.writeValueAsString 安全构建 JSON

### 问题 7: SessionManager 并发不安全
- **现象**: ArrayList 在多线程 SSE 回调下可能抛 ConcurrentModificationException
- **原因**: ArrayList 非线程安全
- **解法**: 改用 CopyOnWriteArrayList

### 问题 8: SseEventEmitter 内存泄漏
- **现象**: 客户端异常断开时 emitter 可能永远不被清理
- **原因**: 无定时清理机制
- **解法**: 添加 @Scheduled 定时任务，每 60 秒清理超过 10 分钟的 stale emitter

### 问题 9: PlanningAgent normalizeJson 脆弱
- **现象**: 硬编码的字符串替换无法覆盖 LLM 所有输出变体
- **原因**: 逐个替换 "ATTRACTIONS"、"hotels" 等，遗漏其他变体
- **解法**: 用正则表达式 + Matcher.appendReplacement 统一处理大小写和单复数

### 问题 10: ChatRequest 缺少验证 + GlobalExceptionHandler 魔法数字
- **现象**: ChatRequest 无输入验证，错误码用魔法数字判断
- **解法**: 添加 @NotBlank 注解和手动验证，switch 改用枚举名比较

## 2026-05-31 - RAG 知识库功能实现

### 问题
项目缺少 RAG 知识库功能，导致 Agent 无法访问专业知识，旅行规划建议不够专业和准确。

### 原因
- 原有系统仅依赖 LLM 的通用知识
- 缺少旅行领域的专业知识库
- 没有向量检索能力

### 解法
实现了完整的 RAG 知识库功能：

1. **技术选型**：
   - 向量数据库：pgvector（复用现有 PostgreSQL）
   - Embedding 模型：千问 text-embedding-v4（阿里云百炼 API）
   - 文档处理：Spring AI DocumentReader

2. **实现内容**：
   - 添加 pgvector、OpenAI、Tika 依赖
   - 配置 pgvector 和千问 Embedding
   - 创建 TripKnowledge 实体和仓库
   - 实现 EmbeddingService 文本向量化服务
   - 实现 DocumentSplitterService 文档分块服务
   - 实现 KnowledgeService 知识管理服务
   - 实现 RagService RAG 增强服务
   - 实现 KnowledgeController REST API
   - 集成到 PlanningAgent
   - 创建预置知识导入服务

3. **关键配置**：
   ```yaml
   spring:
     ai:
       openai:
         base-url: https://dashscope.aliyuncs.com/compatible-mode/v1
         api-key: ${QWEN_API_KEY:}
         embedding:
           options:
             model: text-embedding-v4
       vectorstore:
         pgvector:
           initialize-schema: true
           index-type: HNSW
           distance-type: COSINE_DISTANCE
           dimensions: 2048
           table-name: trip_knowledge
   ```

4. **API 接口**：
   - POST `/api/knowledge/upload` - 上传知识文档
   - GET `/api/knowledge/search` - 搜索相关知识
   - GET `/api/knowledge/search/category` - 按类别搜索
   - GET `/api/knowledge/ask` - RAG 问答
   - GET `/api/knowledge/stats` - 获取统计信息
   - DELETE `/api/knowledge/{id}` - 删除文档

5. **预置知识**：
   - 景点：故宫、长城、西湖、九寨沟、张家界
   - 交通：高铁、首都机场、浦东机场等
   - 签证：过境免签、签证申请等
   - 美食：北京烤鸭、四川火锅、广州早茶等

### 验证方式
1. 启动应用，检查日志中是否显示"预置旅行知识数据完成"
2. 调用 `/api/knowledge/stats` 查看知识库统计
3. 调用 `/api/knowledge/search?query=北京故宫` 验证搜索功能
4. 调用 `/api/knowledge/ask?question=北京有哪些著名景点` 验证 RAG 问答
5. 发送旅行请求，验证 PlanningAgent 是否使用知识库信息

### 注意事项
1. 需要配置 `QWEN_API_KEY` 环境变量
2. PostgreSQL 需要启用 pgvector 扩展
3. 向量维度为 2048（text-embedding-v4 标准）
4. 文档分块大小默认 500 字符

## 2026-05-31 - 意图识别与边界处理实现

### 问题
1. 所有用户消息都走完整的 Plan-and-Execute 流程，即使只是问天气
2. 多轮对话中，用户修改请求时 AI 可能不理解上下文
3. 不合理请求没有处理机制
4. 工具调用失败时没有降级策略

### 解法

#### 1. 意图识别

**新增文件**：
- `service/intent/UserIntent.java` - 意图枚举
- `service/intent/IntentDetectionService.java` - 意图识别服务
- `service/intent/RequestValidationService.java` - 请求验证服务

**修改文件**：
- `agent/TripAgent.java` - 集成意图识别，根据意图选择处理流程

**意图类型**：
- `TRAVEL_PLAN` - 旅行规划请求（完整行程规划）
- `WEATHER` - 天气查询
- `ATTRACTION` - 景点查询
- `RESTAURANT` - 美食/餐厅查询
- `HOTEL` - 酒店/住宿查询
- `MODIFY_PLAN` - 修改已有计划
- `CHAT` - 闲聊

**处理流程**：
```
用户消息 → 意图识别 → 
  - TRAVEL_PLAN → Plan-and-Execute 完整流程
  - WEATHER/ATTRACTION/RESTAURANT/HOTEL → 直接调用工具
  - MODIFY_PLAN → 根据上下文调整计划
  - CHAT → 直接回复
```

#### 2. 多轮对话上下文理解

**修改文件**：
- `agent/planning/PlanningAgent.java` - 优化提示词和上下文

**提示词优化**：
- 添加规则：根据上下文调整计划，而不是重新生成
- 添加规则：注意分析历史对话，理解用户的真实意图
- 添加规则：如果用户只是询问信息，直接回答

**buildUserMessage 优化**：
- 添加历史对话上下文，让 LLM 理解之前的对话

#### 3. 不合理请求处理

**新增文件**：
- `service/intent/RequestValidationService.java` - 请求验证服务

**验证规则**：
- 旅行规划请求：必须包含城市和天数
- 天气/景点/餐厅/酒店查询：必须包含城市

#### 4. 工具调用失败处理

**修改文件**：
- `agent/core/ToolRegistry.java` - 添加重试和降级机制

**重试机制**：
- 最大重试次数：3 次
- 重试间隔：1s、2s、3s（指数退避）

**降级策略**：
- 重试失败后返回友好提示
- 不同工具返回不同的降级消息

### 验证方式

1. **意图识别测试**：
   - 输入"帮我规划北京3日游" → TRAVEL_PLAN
   - 输入"北京天气怎么样" → WEATHER
   - 输入"北京有什么好吃的" → RESTAURANT
   - 输入"缩减到三天" → MODIFY_PLAN

2. **多轮对话测试**：
   - 第一轮："帮我规划北京3日游"
   - 第二轮："缩减到两天" → 应该调整计划

3. **不合理请求测试**：
   - 输入"我想去旅游" → 应该询问城市和天数
   - 输入"北京天气" → 应该询问具体城市

4. **工具调用失败测试**：
   - 模拟 API 超时 → 应该重试 3 次
   - 重试失败 → 应该返回友好提示

### 注意事项
1. 意图识别优先使用关键词，失败时使用 LLM
2. 多轮对话依赖 chatHistory 的正确传递
3. 请求验证是可选的，可以配置是否启用
4. 工具重试会增加延迟，需要权衡

## 2026-06-01 - MCP 客户端连接问题修复 (PrematureCloseException)

### 问题
MCP 客户端连接出现 `PrematureCloseException: Connection prematurely closed BEFORE response` 错误。

### 原因
1. Spring AI 2.0.0-M8 的 Streamable HTTP transport 存在已知 bug (GitHub Issue #3763)
2. Reactor Netty 客户端连接池管理问题
3. 即使 MCP 服务器在本地运行，仍通过 HTTP 网络通信

### 解法
改用 STDIO transport，实现进程内通信：

1. **trip-tools-server 修改**：
   - 移除 `spring-ai-starter-mcp-server-webmvc` 依赖
   - 添加 `spring-ai-starter-mcp-server` 依赖 (STDIO 模式)
   - 配置 `spring.main.web-application-type: none` 禁用 Web 服务器

2. **trip-agent 修改**：
   - 配置从 `streamable-http` 改为 `stdio`
   - 使用 `cmd.exe /c java -jar` 启动 MCP 服务器子进程

### 关键配置

**trip-agent application.yml**:
```yaml
spring:
  ai:
    mcp:
      client:
        stdio:
          connections:
            trip-tools:
              command: cmd.exe
              args:
                - /c
                - java
                - -jar
                - trip-tools-server/target/trip-tools-server-1.0.1.jar
```

**trip-tools-server application.yml**:
```yaml
spring:
  main:
    web-application-type: none  # 禁用 Web 服务器
```

### 优势
1. ✅ 无网络通信，避免 PrematureCloseException
2. ✅ 进程内通信，更稳定
3. ✅ 保留 reasoningContent 支持 (DeepSeek 思考字段)
4. ✅ 无需降级 Spring AI 版本

### 注意事项
1. Windows 需要使用 `cmd.exe /c` 包装 Java 命令
2. MCP 服务器作为子进程启动，生命周期由客户端管理
3. 请求超时从 30s 增加到 60s，适应 STDIO 启动时间

## 2026-06-01 - RAG 知识库功能实现（简化版）

### 问题
项目需要 RAG 功能来展示技术能力，用于简历和面试。

### 解法
实现了基于 pgvector + 千问 Embedding 的 RAG 知识库：

1. **技术选型**：
   - 向量数据库：pgvector（复用 PostgreSQL）
   - Embedding 模型：千问 text-embedding-v4（1024 维）
   - 文档格式：Markdown 单文件

2. **实现内容**：
   - 添加 `spring-ai-starter-vector-store-pgvector` 和 `spring-ai-starter-model-openai` 依赖
   - 实现 `KnowledgeService`：MD 解析、分块、向量化、检索
   - 集成到 `PlanningAgent`，规划时自动注入本地人攻略
   - 添加南京本地人攻略知识文档

3. **关键配置**：
   ```yaml
   spring:
     ai:
       openai:
         base-url: https://dashscope.aliyuncs.com/compatible-mode/v1
         api-key: ${QWEN_API_KEY}
         embedding:
           options:
             model: text-embedding-v4
             dimensions: 1024
       vectorstore:
         pgvector:
           initialize-schema: true
           index-type: HNSW
           distance-type: COSINE_DISTANCE
           dimensions: 1024
           table-name: trip_knowledge_v2
   ```

4. **RAG 流程**：
   - 启动时加载 `nanjing-local.md` → 按 `##` 标题分块 → 向量化存储
   - 规划时：用户消息 → 向量检索 Top3 → 注入 Prompt → 生成计划

5. **测试结果**：
   - 知识加载：8 个分块成功向量化
   - RAG 检索：AI 回复包含本地人攻略（芳婆糕团店、李记清真馆等）
   - 避坑提示：自动过滤游客店

### 验证方式
1. 启动应用，日志显示"知识文档加载完成，共 8 个分块"
2. 发送"南京有什么好吃的"，验证 AI 是否引用攻略内容

### 注意事项
1. 需要配置 `QWEN_API_KEY` 环境变量
2. PostgreSQL 需要安装 pgvector 扩展
3. HNSW 索引限制 2000 维度，使用 1024 维
